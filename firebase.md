# Firebase Firestore Documentation

## Overview

This document describes the Firebase Firestore database structure for the Crew Career enrollment system. The database uses document-based storage with UUID-generated document IDs and supports real-time data synchronization for pilot career tracking.

## Collection Structure

### `users` Collection

The primary collection for storing pilot information and career progression data. Each document represents a unique pilot enrolled in the career mode system.

#### Document ID Format
- **Type:** Auto-generated UUID (e.g., `"H3yijAKn27VBZCOmfetunsLxUTG3"`)
- **Generated by:** Firebase SDK using `.add()` method
- **Purpose:** Unique identifier for each pilot document

#### Field Schema

| Field Name | Data Type | Description | Required | Default Value |
|------------|-----------|-------------|----------|---------------|
| `activeEvent` | Dictionary | Currently active event data | No | `{}` |
| `baseAirport` | String | Pilot's assigned base airport ICAO code | Yes | - |
| `callsign` | String | Unique pilot identifier used in system | Yes | - |
| `createdAt` | Timestamp | Document creation timestamp | Yes | Current time |
| `createdBy` | String | Creator identifier (usually system) | Yes | Callsign |
| `currentLocation` | String | Current airport location (ICAO code) | Yes | Base airport |
| `earnings` | Number | Total career earnings in currency units | Yes | `0` |
| `flightHours` | Number | Total flight hours accumulated | Yes | `0` |
| `lastLogin` | Timestamp | Last login timestamp | No | `null` |
| `name` | String | Pilot's full name | Yes | From Neon DB |
| `rank` | String | Current pilot rank/position | Yes | `"Cadet"` |
| `role` | String | System role (pilot, admin, etc.) | Yes | `"pilot"` |
| `totalFlights` | Number | Total number of flights completed | Yes | `0` |
| `typeRatings` | Array[String] | List of aircraft type ratings | Yes | `[]` |

#### Sample Document Structure

## Database Operations

### 1. User Enrollment Process

The enrollment process involves multiple steps across Firebase and Neon DB:

#### Step 1: Authentication Verification
```javascript
const session = await auth();
if (!session?.user?.callsign) {
    throw new Error("Authentication required");
}
```

#### Step 2: Data Validation
```javascript
const { baseAirport, typeRating } = await request.json();
if (!baseAirport || !typeRating) {
    throw new Error("Base airport and type rating are required");
}
```

#### Step 3: User Data Retrieval (Neon DB)
```javascript
const userData = await db
    .select({ ifcName: users.ifcName })
    .from(users)
    .where(eq(users.id, callsign))
    .limit(1);
```

#### Step 4: Firebase User Check
```javascript
const usersQuery = firebaseDb.collection('users').where('callsign', '==', callsign);
const usersSnapshot = await usersQuery.get();
```

#### Step 5: Create or Update User Document

**For New Users:**
```javascript
await firebaseDb.collection('users').add({
    callsign: callsign,
    name: userName,
    baseAirport: baseAirport,
    typeRating: typeRating,
    careerMode: true,
    enrolledAt: new Date().toISOString(),
    rank: 'Cadet',
    flightHours: 0,
    totalFlights: 0,
    earnings: 0,
    role: 'pilot',
    createdAt: new Date().toISOString()
});
```

**For Existing Users:**
```javascript
await userDoc.ref.update({
    baseAirport: baseAirport,
    typeRating: typeRating,
    careerMode: true,
    enrolledAt: new Date().toISOString()
});
```

## Data Relationships

### Firebase ↔ Neon DB Integration

1. **Neon DB (Primary User Data):**
   - Stores core user authentication data
   - Contains `ifcName` field used for Firebase name population
   - Uses callsign as primary identifier

2. **Firebase Firestore (Career Data):**
   - Stores career-specific information
   - Maintains real-time career progression data
   - Provides fast queries for career-related operations

### Data Flow

```
User Login → Neon DB (Authentication) → Firebase (Career Data)
     ↓              ↓                        ↓
  Callsign    →  ifcName  →  Firebase Name
Session Data →  Email, etc. →  Firebase Fields
```

## Security Rules

### Collection-Level Rules
```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId} {
      // Allow read access to authenticated users for their own data
      allow read: if request.auth != null && 
                   resource.data.callsign == request.auth.token.callsign;
      
      // Allow write access only through server-side operations
      allow write: if false; // All writes go through Cloud Functions
    }
  }
}
```

### Index Requirements

The following composite indexes should be created for optimal query performance:

1. **Enrolled Users by Base Airport:**
   - Fields: `baseAirport` (asc)

2. **Leaderboard Queries:**
   - Fields: `flightHours` (desc)
   - Fields: `totalFlights` (desc)
   - Fields: `earnings` (desc)

3. **User Lookup by Callsign:**
   - Fields: `callsign` (asc)

## Error Handling

### Common Error Scenarios

1. **Authentication Errors**
   ```javascript
   if (!session?.user?.callsign) {
       return NextResponse.json(
           { error: "Authentication required" },
           { status: 401 }
       );
   }
   ```

2. **User Not Found (Neon DB)**
   ```javascript
   if (userData.length === 0) {
       return NextResponse.json(
           { error: "User not found in database" },
           { status: 404 }
       );
   }
   ```

3. **Already Enrolled**
   ```javascript
   if (userData.baseAirport) {
       return NextResponse.json({
           success: true,
           alreadyEnrolled: true,
           message: "User already enrolled in career mode"
       });
   }
   ```

4. **Database Errors**
   ```javascript
   try {
       // Firebase operations
   } catch (error) {
       console.error('Enrollment API error:', error);
       return NextResponse.json(
           { error: "Enrollment failed", details: error.message },
           { status: 500 }
       );
   }
   ```

### Schema Evolution

When adding new fields to the user schema:

1. **Backward Compatibility:**
   - Always provide default values for new fields
   - Use optional fields when possible
   - Test migration scripts thoroughly

2. **Migration Script Example:**
   ```javascript
   // Add new field to existing documents
   const snapshot = await firebaseDb.collection('users').get();
   const batch = firebaseDb.batch();
   
   snapshot.forEach(doc => {
       const ref = doc.ref;
       batch.update(ref, {
           newField: 'defaultValue',
           updatedAt: new Date().toISOString()
       });
   });
   
   await batch.commit();
   ```

## Best Practices

### Performance Optimization

1. **Query Optimization:**
   - Use compound queries with appropriate indexes
   - Limit result sets with `.limit()`
   - Use pagination for large datasets

2. **Document Design:**
   - Keep documents under 1MB
   - Avoid deeply nested arrays
   - Use subcollections for large arrays

3. **Real-time Updates:**
   - Use `onSnapshot` for live career progression
   - Implement offline persistence
   - Handle connection state changes

### Security Considerations

1. **Server-Side Operations:**
   - All Firebase writes go through server routes
   - Client-side only reads authenticated user's data
   - No direct Firebase access from client

2. **Data Validation:**
   - Validate all inputs on server
   - Sanitize string inputs
   - Validate data types before writing

3. **Access Control:**
   - Role-based access for admin functions
   - Session-based authentication
   - Audit logging for sensitive operations

## Troubleshooting

### Common Issues

1. **"Permission Denied" Errors:**
   - Check security rules configuration
   - Verify authentication tokens
   - Ensure server-side operations only

2. **Slow Query Performance:**
   - Check for missing indexes
   - Optimize compound queries
   - Consider denormalization for frequent queries

3. **Data Inconsistency:**
   - Implement transaction safety
   - Use batch operations for related updates
   - Add data validation checks

4. **Connection Issues:**
   - Implement retry logic with exponential backoff
   - Handle offline scenarios gracefully
   - Monitor connection state

## API Reference

### Enrollment Endpoint

**Endpoint:** `POST /api/crewcareer/enroll`

**Request Body:**
```json
{
  "baseAirport": "VABB",
  "typeRating": "A320"
}
```

**Success Response (200):**
```json
{
  "success": true,
  "alreadyEnrolled": false,
  "message": "Enrollment successful"
}
```

**Already Enrolled Response (200):**
```json
{
  "success": true,
  "alreadyEnrolled": true,
  "message": "User already enrolled in career mode"
}
```

**Error Responses:**
- `401`: Authentication required
- `400`: Missing required fields
- `404`: User not found in database
- `500`: Enrollment failed
